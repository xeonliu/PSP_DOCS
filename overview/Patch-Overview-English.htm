Patch Overview

1 Patch System for the PSP™ System

1. 1 About Patches

The term "patch" refers to the procedure for adding or replacing program or data files to extend the functionality of an application, and the new program or data files that are added or replaced during that procedure.
A patch system is provided that can add or replace any program or data file of a previously released application supplied on a UMD™, or of a PSP™ downloadable game application installed on a Memory Stick™.

Note
If you wish to use patches to extend the functionality of your application, contact the SCE Publisher Relations Department during the planning stage.

The main components of the PSP™ patch system are as follows.
  • Patch server
    Server for registering patches and distributing patches over the network.
  • Update feature
    Feature that checks the patch server for new patches and downloads and installs those that it finds.
  • Patch launch feature
    Feature that launches a newly installed program file when starting up a UMD™ or downloadable game.
For the update feature, the system software performs all processes, from checking the patch server to downloading and installing available patches. A feature is also provided that enables an application to use the game update library to check the patch server.
The patch launch feature is required in order to patch the boot program file of a UMD™ game or downloadable game. Because the boot program supplied with a UMD™ game or downloadable game cannot be overwritten, the patch is stored in a specific directory on the Memory Stick™. When the system software launches the UMD™ game or downloadable game, it checks whether a patch exists. If one does exist, the system software launches the updated application by launching the boot program file of the patch rather than the boot program in the UMD™ game or downloadable game.

1. 2 Patch Server

Title Patch Publishing System
SCE provides a patch server for managing and distributing patches.
The Title Patch Publishing System is provided as a special-purpose tool for performing such tasks as uploading a patch package to the development server, submitting it to QA, and promoting it to the production server. This tool is a PlayStation®Network Server Management Tool (web application), which authenticates users according to their PSP™ Developer Network website accounts.
For details about the Title Patch Publishing System, refer to the "PlayStation®Network SMT Portal Users Guide" and the "PlayStation®Network SMT Title Patch Publishing System Users Guide."



Figure 2 Title Patch Publishing System


Patch package
A patch generally consists of multiple files. However, when registered on the patch server a patch package must be created by adding content information such as PARAM.SFO and ICON0.PNG to those files.
A patch package is a type of NPDRM package. For details, see the chapter "Creating a Patch Package."

Patch identifier
Patches are managed for each DISC_ID in PARAM.SFO. If a title consists of multiple discs each with its own DISC_ID, patches must be created and registered separately on the patch server for each one. However, if you want to share one patch among different DISC_IDs for a multidisc title, you can add a GAMEDATA_ID parameter to the PARAM.SFO file of each disc that is to share the patch. If GAMEDATA_ID is defined in PARAM.SFO, then it will be used as the identifier instead of DISC_ID. In other words, GAMEDATA_ID must have the same value as that specified for DISC_ID in PARAM.SFO of the patch.

Patch version
The application and patch versions are managed by the APP_VER parameter in PARAM.SFO.
The version number, which consists of four decimal digits, is conveniently divided into a two-digit major version and a two-digit minor version. A larger number indicates a newer version.

1. 3 Updating

Updating is a process that sends an inquiry to the patch server to check whether a new patch package has been published, and if so, downloads and installs it on the Memory Stick™.

For details about system behavior during updates, see the chapter "Update Process."

1. 4 Patch Launching

When a UMD™ game or downloadable game is launched, system software checks whether a patch has been installed on the Memory Stick™. If a patch has been installed, system software determines whether the boot program in the UMD™ game / downloadable game or the boot program in the patch is newer and starts whichever is newer. System software also determines whether it needs to be updated at this time as well. If so, it prompts the user to update it. For details about this process, see the chapter "Patch Launching Process."

2 Update Process

This chapter describes the update process, which consists of checking whether a new patch has been registered on the patch server and then downloading and installing it.

2. 1 Update Process

During the update process, system software downloads and installs a patch.
To start the update process, open the Options menu from system software by clicking the triangle button for the icon of the game to be updated, and select "Update." System software connects to the network using infrastructure mode and checks the patch server for any new patches and their versions.
If a new patch has been registered, a dialog prompting the user to perform the update will appear.

If the user selects "Yes," system software will download and install the patch package.
If new system software is needed to launch the patch, a dialog will appear that prompts the user to update system software before downloading and installing the patch package.

2. 2 Checking for Patches from an Application

An application can use the game update library APIs at any time to check for new patches. To perform this operation, the PSP™ system must be connected to a network in infrastructure mode. For details, refer to the "Game Update Library Overview" and "Game Update Library Reference."

3 Patch Launching Process

This chapter describes the process used by system software to determine and launch a UMD™ game or downloadable game program that was updated by a patch. It also describes the system update process (which determines whether the system software version meets the minimum application requirements and prompts the user to perform any necessary updates), because this process occurs at the same time.

3. 1 Parameters Indicating the Software Version

The following two parameters represent version information and are used for patches and system updates.
  • PSP_SYSTEM_VER
    This indicates the system software version required to run the application.
  • APP_VER
    This indicates the version of the application or patch itself.
These versions are independent and have no relationship to one another.
Both types of information are specified in PARAM.SFO.

3. 2 Launching Sequence

First, this section explains the behavior when a UMD™ game or downloadable game is launched when a patch has not been installed. In this case, only processing related to system updates is performed, as shown below, based on the PSP_SYSTEM_VER value that was obtained from PARAM.SFO of the UMD™ or downloadable game.



Figure 5 Launch Sequence


  (1)     The user selects (launches) the icon of the UMD™ game or downloadable game.  
  (2)     PSP_SYSTEM_VER is obtained from PARAM.SFO. This value is the version of the   
  system software required by the application.  
  (3)     The system software version is compared with the PSP_SYSTEM_VER value that   
  was obtained.  
  (4)     If the system software version is greater than or equal to PSP_SYSTEM_VER, the

  (5)     If the system software version is less than PSP_SYSTEM_VER, the user is

 (9)   If the UMD™ game or downloadable game version is newer than the patch version,  
 the boot file of the UMD™ game or downloadable game is launched. 
         Note: 
 Remastering may create a UMD™ or downloadable game version that is  
 newer than the patch version. 
 (10)   If the game version and patch version are the same or the patch version is newer,  
 the patch boot file is launched (patch launch). 

4 Creating an Application that Supports Patching

This chapter describes matters that you should carefully consider when creating an application that will use patches for extending its functionality.

4. 1 Overview of the Process

The process for developing and releasing an application that supports patching is briefly described below.

(1) Approving plans

If you want to create an application for which you plan to use patches to extend its functionality, contact the SCE Publisher Relations Department during the planning stage to discuss the content of the patches, estimated time periods, and estimated number of times patches will be made.

(2) Creating and testing the application

Generally speaking, there are no special requirements for supporting patches in an application. However, if your application will use the game update library to check for new patches, you must test this functionality. Create a dummy patch package, use the Title Patch Publishing System to upload it to the development patch server, and verify that update processing works as intended.
When creating the master disc, set APP_VER appropriately in PARAM.SFO. Also, if you want to share the patch for an application that has multiple master discs, specify the representative DISC_ID for GAMEDATA_ID.

(3) Creating and testing the patch package

When releasing a patch, create the patch program and data, and create and test the patch package together with content information. For information about the procedure for creating a patch package, refer to the "PSP™ Patch Package Creation Guide" document.

(4) Testing using the development patch server

Use the Title Patch Publishing System to upload the patch package to the development patch server. If all tests thus far have been successful, there should be no problems, but be sure to test the update itself and the updated application.
In particular, verify that the update can be performed properly as intended from all previously released versions.

Note

To test patch distribution using the development patch server, check the Title Patch Publishing System setting "Server Setting (development server)" -> "ACL Setting (development server)" to verify that the IP address of the machine from which the application will be run is permitted to access the server.
If access is not permitted, the application will be considered not to have an update.

(5) Submitting to QA

Use the Title Patch Publishing System to submit the patch package to QA. You can also submit multiple patch packages with different versions at the same time.

(6) Releasing the patch

When the patch package has been approved by QA, it can be moved to the production patch server. QA will perform tasks for releasing the package. If you have a desired release date, contact the SCE Publisher Relations Department.

4. 2 Making use of Versions

The application version is represented by the APP_VER value "xx.yy" in PARAM.SFO. xx represents the major version and yy represents the minor version.
Assume the master disc or package version is "01.00." Basically the minor version is incremented by 1 each time a patch is released ("01.01," "01.02," "01.03," …). The major version is incremented by 1 and the minor version is returned to 0 only when the application is remastered ("02.00," "03.00," "04.00," …).
System software treats the version as a 4-digit number without the decimal point, and it recognizes a larger number as a newer version. Also, if APP_VER does not exist in PARAM.SFO, system software considers it to be "00.00."

Version transitions proceed sequentially and linearly from the oldest version to the newest version. When a version is updated, the result must be the same regardless of which version of the game was used as the base version for the update.

4. 3 Patch Package Configurations

The two types of patch packages, cumulative patch packages and differential patch packages, have different configurations.

Cumulative patch package
A cumulative patch package can update an application to the patch version from any past version. It contains all additional data released in the past and required for the patch version.
When the latest version is a cumulative patch package, only that package is placed on the patch server, because that patch package alone can be used to update the application from any past version.



Figure 8 Cumulative Patch Package


Differential patch package
A differential patch package can update an application to the patch version but only from a certain specific version. Of the additional data required for the patch version, the differential patch package only contains the data not included in that specific original version. The PARAM.SFO file contained in the differential patch package must contain TARGET_APP_VER, which indicates the differential target version, in addition to APP_VER, which indicates its own version.



Figure 9 Differential Patch Package


When the latest version is a differential patch package, all patch packages that are required for updating the application to the differential target version are placed on the patch server in addition to that differential patch package. System software automatically selects, downloads, and installs the required patch packages from the current application version.

Patch package configuration examples
Let's consider an example in which three patches (versions 1.01, 1.02, and 1.03) were released for a UMD™ game or downloadable game.
  • 1.01 (cumulative patch package)
    First patch package. The first patch package is handled as a cumulative patch package.
  • 1.02 (differential patch package)
    Differential package from 1.01. TARGET_APP_VER is set to 01.01 and this package contains only files that were added or changed since version 1.01.
    Two patch packages (1.01 and 1.02) are placed on the patch server, and only the 1.02 patch is downloaded and installed if the user had already applied the 1.01 patch, while both the 1.01 and 1.02 patches will be downloaded and installed if the user had not applied the 1.01 patch.
  • 1.03 (differential patch package)
    Differential package from 1.02. TARGET_APP_VER is set to 01.02 and this package contains only files that were added or changed since version 1.02.
    Three patch packages (1.01, 1.02, and 1.03) are placed on the patch server, and only the required packages are downloaded and installed depending on the application version the user is using.



Figure 10 Patch Package Configuration Example


4. 4 Support When Application is Remastered

This section uses an example to explain how to assign application versions when both patching and remastering an application. In the following example, we consider the case of remastering the application after cumulative patch 1.01 and differential patch 1.02 are applied.
First, each time the patch is updated, the minor version is incremented by 1 as described earlier: "01.01," "01.02," …
Then when the application is remastered, the major version is updated to "02.00." (To remaster the application three or more times, update the major version to "03.00," "04.00," …)
At the same time, prepare a differential patch for providing the equivalent content as the remastered application. Set the value of APP_VER for this differential patch to "02.00", which is the same version as the remastered application (see Figure 11).

When differential patches are released again after the application is remastered, set APP_VER to "02.01" and "02.02" (see Figure 12). Note that the post-remaster patches are launched from both the original master disc and the remaster disc.

5 Creating a Patch Package

This chapter describes the main processing performed by a program launched as a patch and explains how to create PARAM.SFO, PBOOT.BIN, and other files distributed in the patch package. For a list of required files and the procedure for collecting them in a package, refer to the "PSP™ Patch Package Creation Guide" document

5. 1 Patch Program Processing

Normally, the processing performed by PBOOT.BIN when it is launched as a patch, is virtually the same as that performed by EBOOT.BIN of a UMD™ game or downloadable game. When the game is launched by PBOOT.BIN, data files and program files in the UMD™ game or downloadable game (other than EBOOT.BIN) can be read from the UMD™ or downloadable game image in the same way as if the game were launched from the disc. When the patch contains revised versions of those files, the game can access them from the directory where the patch was installed. (An alternative way of explaining this is that when program files other than EBOOT.BIN or data files are revised, they are distributed together with the revised PBOOT.BIN file so that they can be read from the directory where the patch was installed.)

5. 3 Creating a Patch Program File

When developing a program to be included in a patch, be sure to test not only that the program itself works properly, but also that the application works properly after the old version is updated by the program.
You can recreate the state when the patch is installed by unzipping the ZIP package obtained from the Simple Authoring Service in the root directory of a Memory Stick™, and then you can perform operational testing using the PSP™ development tool.

5. 4 Creating Patch Content Information Files and Data Files

A patch package is encrypted. After it is successfully installed, it will be decrypted and the contents verified to be correct. Therefore, no specific means of protecting it from tampering is required. However, it is recommended that the application perform its own encryption processing for important files.
