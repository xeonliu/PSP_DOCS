Internet Browser Utility Overview

1 About the Utilities

For an overview of the utilities, refer to the "Utility Services Overview" document.

2 Internet Browser Utility Overview

The Internet browser utility provides a means of calling the Internet browser from a game program.
An Internet browser that is called from a game program, which has most of the same functions as an Internet browser that is called from the home menu, can be used for browsing game-related sites or for downloading data. For information about restrictions in an Internet browser that is called from a game program, see "Internet Browser Utility Restrictions." The Internet browser utility also provides functions that support its use from a game program such as being able to specify user interface restrictions or receiving return values from the content. For details, see the corresponding sections in this document and refer to the Utility_InternetBrowser-Reference.

2. 2 Operation During a Suspend

The Internet browser utility differs from the other utilities in that it is not terminated when a suspend occurs. When a resume occurs, the page that had been displayed when the suspend occurred will continue to be displayed.

2. 3 Required Memory

To run the Internet browser utility, at least 5M bytes (at least 6M bytes if the
Macromedia® Flash® Player function is enabled) must be free in the user memory space as memory required for loading modules and as a stack area, and of that free area, 3.5M bytes (4.5M bytes if the Flash® Player function is enabled) must be in a contiguous area.
In addition to this area, a recommended memory area of at least 8M bytes (4M bytes minimum) must be provided as an Internet browser-specific heap. Since cache memory is also allocated from the Internet browser-specific heap, this should be taken into account when specifying the heap.

For the stack size of the internal system thread that is specified in sceNetApctlInit(), add at least 16KB to the minimum stack size, which is indicated by SCE_NET_APCTL_LEAST_STACK_SIZE.

Example

 
#define APCTL_PRIO			48
#define APCTL_HANDLER_STACKSIZE		(1024 * 16)
#define APCTL_STACKSIZE		(SCE_NET_APCTL_LEAST_STACK_SIZE + \
					 APCTL_HANDLER_STACKSIZE)

int ret = sceNetApctlInit(APCTL_STACKSIZE, APCTL_PRIO);

3 Internet Browser Utility Support Functions

The Internet browser utility has functions for supporting its use from a game program in addition to the functions that are available when the Internet browser is called from the home menu.
This section describes these support functions.

3. 1 Specifying the User Interface Level

In addition to the mode in which all user interface functions are available, the Internet browser utility has modes in which some functions are limited or no user interface is displayed at all. Using these modes, browsing can be limited to the page specified in the initialization parameters, or browsing can be more naturally connected to the game program.

In limited-user-interface mode, the icons or menu items of limited functions are grayed out and can no longer be selected.
The limited user interface functions are as follows.
  • Entering and opening a URL
  • Bookmark function
  • Visited page history function
  • Home page setting
  • Image display ON/OFF
    Runs with image display ON regardless of the setting.
  • Animation ON/OFF
    Runs with animation ON regardless of the setting.
  • JavaScript ON/OFF
    Runs with JavaScript ON regardless of the setting.
  • Flash ON/OFF
    Runs with Flash ON regardless of the setting. However, the user must have already set "Enable Flash® Player" from the system settings and must have specified "Enable Flash® Player function" for an initialization parameter when starting up the utility.
  • Memory conservation ON/OFF
    Runs with Memory conservation OFF regardless of the setting
  • Connection setting
    This is controlled by a temporary setting that is passed in an initialization parameter.
  • Proxy server setting
  • Authentication information deletion
  • Cookie reception setting
    This is controlled by a temporary setting that is passed in an initialization parameter.
  • Cookie deletion
  • Input history deletion
  • Cache size setting
    This is controlled by a temporary setting that is passed in an initialization parameter.
  • Cache deletion

In no-user-interface mode, in addition to the function limitations in limited-user-interface mode, the menu display itself is disabled.
However, the cursor and tab indicator displays are not automatically disabled. To disable them, you must specify that they be disabled in separate initialization parameters.

For details about how to set these modes, refer to the description of SceUtilityHtmlViewerParam in the Utility_InternetBrowser-Reference.

3. 2 Changing Temporary Settings

When the Internet browser utility is called with a limited user interface or no user interface, some browser settings such as the cookie reception setting can be temporarily changed. When the utility is called with a full user interface, the settings cannot be temporarily changed.

The setting items that can be temporarily changed are as follows.
  • Cookie reception setting
    Changing the cookie reception setting enables the behavior when a cookie is received to match the behavior of a site related to the game program.
    However, even though the setting is temporary, the received cookie is not unconditionally deleted when the utility is shut down.
  • Cache size
    The cache size is used for specifying the size of the libhttp simple cache mechanism in memory. Temporarily changing the cache size enables the balance between the memory required for file acquisition efficiency and for browsing to be optimized for a site related to the game program.
    Since cache memory is also allocated from the Internet browser-specific heap, this should be taken into account when specifying the cache size.
  • Home page
    Temporarily changing the home page enables the browser to return to the specified page by selecting the home icon regardless of the browser setting.
  • Text size
    Small, normal, or large can be specified for the text size that is used for display regardless of the browser setting.
  • Display mode
    Normal, just fit, or smart fit can be specified for the layout method that is used for display regardless of the browser setting.
  • Connection mode
    Automatic, manual (first connection only), or manual (every connection) can be specified for the connection setting selection method when the network connection is established regardless of the browser setting.

For details about how to set these items, refer to the description of SceUtilityHtmlViewerConfig in the Utility_InternetBrowser-Reference.

3. 3 Registering Bookmarks

A bookmark can be registered in the Internet browser from a game program. A bookmark is registered by calling the Internet browser utility with bookmark information set in an initialization parameter.
Since no function is provided for checking whether or not the specified bookmark is already registered, duplicate registrations should be prevented by using a scheme such as saving registration results in save data so operation can be changed the next time the browser is used. A user confirmation dialog is also displayed when a bookmark is registered in the Internet browser.
Since there is no guarantee that the bookmark can be registered accurately and since a bookmark can also be deleted by the user, this function should not be used for any use that is indispensable for game progress.

The following information can be registered for a bookmark.
  • URL
  • Caption

For details about how to set this item, refer to the description of SceUtilityHtmlViewerBookmark in the Utility_InternetBrowser-Reference.

3. 4 Specifying the Upload/Download Directory/File Name

The Internet browser utility provides a function for supporting the exchange of files with a website by fixing the directory name and file name when uploading or downloading a file.
By specifying a valid name for the directory name, you can restrict the directory that can be specified when uploading or downloading to that directory.
Similarly, by specifying a valid name for the file name, you can fix the file name that is used when uploading or downloading.

For details about how to set these items, refer to the description of SceUtilityHtmlViewerFileParam in the Utility_InternetBrowser-Reference.
For details about downloading, see the section entitled "File Download Function."

3. 5 User Interface Limitations During Downloading

Limitations can be specified for the user interface when using the Internet browser to download a file from a website.

The limitations that can be specified are as follows.
  • Do not display a confirmation dialog when downloading begins
  • Do not display a confirmation dialog when downloading is completed
  • Do not display a save destination confirmation dialog
  • Do not change the save destination directory and file name in the save destination confirmation dialog

For details about operation when limitations related to the save destination confirmation dialog are specified, see the section entitled "File Download Function."

3. 6 Temporary Disabling the Internet Browser Startup Restriction

When the Internet browser startup restriction is enabled in the system settings, startup is restricted as a function of the authentication number input on startup or resume. This applies to the Internet browser utility as well.
However, by setting an initialization parameter, this startup restriction can be temporarily disabled. The disabling continues until the Internet browser utility is shut down.

3. 7 Specifying the URL to be Opened on Startup

You can specify the URL that is to be opened first when the Internet browser utility is started up.
The specification methods are as follows from highest to lower priority.
  1. Specify "Open network connection confirmation page" in an initialization parameter
    A network connection confirmation page, which is used as the connection destination when authentication occurs in a wireless LAN environment, is provided for each language. It is specified when you want to start up the Internet browser for simply confirming the network connection.
  2. Specify the initial connection URL in an initialization parameter
  3. Specify the home page URL in "Temporarily change settings"
If nothing is specified, the home page that was specified in the browser settings is opened on startup in a similar method as when the Internet browser utility is started up from the home menu.

If the Internet browser utility was started up with no user interface specified, the user can press the Triangle button to reload the network connection confirmation page or the page that was specified by the initial connection URL.
This function is provided as a means of opening the target page when page loading fails, the network connection is disconnected during loading, or an unexpected page is opened at a hot spot.
If you do not want the user to use this function, do not specify an initial connection URL and the home page URL will be specified in "Temporarily change settings".

For details about setting these items, refer to the descriptions of SceUtilityHtmlViewerParam and SceUtilityHtmlViewerConfig in the Utility_InternetBrowser-Reference.

3. 8 Specifying the Number of Tabs

The Internet browser utility has a function for running as a tabbed browser with up to three tabs. If you want to change the number of tabs, you can specify the total number of tabs in an initialization parameter.
However, even if you specify 1 for the number of tabs, the tab indicator display will not be automatically disabled. If you want to disable this display, you must specify that it be disabled in a separate initialization parameter.

For details about how to set this item, refer to the description of SceUtilityHtmlViewerParam in the Utility_InternetBrowser-Reference.

3. 9 Automatically Disconnecting the Network Connection

You can use an initialization parameter to specify whether or not to automatically disconnect the network connection when the Internet browser utility is shut down. The following instructions can be specified.
  • Automatically disconnect
  • Do not automatically disconnect
  • Display a dialog to confirm whether or not to disconnect

For details about how to set this item, refer to the description of SceUtilityHtmlViewerParam in the Utility_InternetBrowser-Reference.

3. 10 Specifying Whether Confirmation is Required on Shutdown

You can use an initialization parameter to specify whether or not to display a shutdown confirmation dialog when shutting down the Internet browser utility.

3. 11 Specifying Whether to Display or Hide the Cursor

You can use an initialization parameter to specify whether or not to display the cursor.
However, note that if the cursor is not displayed, the user may be unable to browse as expected when mouse operations are required due to the page configuration, JavaScript, or Flash®.

3. 12 Specifying Whether to Display or Hide the Tab Indicator

You can use an initialization parameter to specify whether or not to display the tab indicator.

3. 13 Specifying Whether or Not to HOLD the Analog Controller

You can use an initialization parameter to specify whether or not to HOLD the analog controller when the HOLD switch is turned on.
Since the HOLD state of the analog controller is entrusted to the game program, this specification can be used to ensure operational consistency.

3. 14 Specifying Whether or Not Flash® Player Functionality is Enabled

You can use an initialization parameter to specify whether or not to enable the Flash® Player function.
Note that if you specify that it should be enabled, the memory required for loading modules is increased by 1M byte.
Also, to actually enable the Flash® Player function, the user must already have set "Enable Flash® Player" in the system settings and Flash® must be set to ON in the browser settings.
If you specify that the Flash® Player function is to be enabled when "Enable Flash® Player" has not been set from the system settings, a dialog indicating that the Flash® Player must be enabled in the system settings is displayed when the utility is started up.

3. 15 Enabling and Disabling the Use of the L/R Buttons for Back/Forward Functions

Whether or not the L/R buttons can be used for the back/forward functions can be specified in an initialization parameter.
Although the L/R buttons can be used to provide back/forward functions even when no user interface mode is selected by the user interface level setting, they can be disabled by this specification to restrict user operations within the content.

3. 16 Getting the Maximum Browser Heap Usage

You can get information indicating the maximum usage of the heap area. The heap area is specified by an Internet browser utility startup parameter. This information should be referenced after the Internet browser utility transitions to SCE_UTILITY_COMMON_STATUS_SHUTDOWN status.

The information that is obtained here can be used as a source for determining the heap size to be provided to the Internet browser, however, if the browser heap size gets too close to the maximum usage, it could cause a significant drop in operating speed. A value that is 10 to 30 percent greater than the maximum usage should be set for the browser heap size.
Cache memory is also allocated from the browser heap, and this memory is included in the value obtained for the maximum usage.

For details about how to set this item, refer to the description of SceUtilityHtmlViewerParam in the Utility_InternetBrowser-Reference.

3. 17 Specifying a Proprietary HTTP Request Header

A proprietary HTTP request header can be sent to the server according to a game program specification.
This enables a site to identify a connection initiated by an Internet browser that was started up from a specific game.

For details about how to set this item, refer to the description of SceUtilityHtmlViewerParam in the Utility_InternetBrowser-Reference.

3. 18 Automatic HTTP Authentication

Authentication information specified by the game program can be used to perform HTTP authentication without making the user enter authentication information.
Although this simplifies processing, if someone else tries to access the site besides the game program, they might be blocked at the HTTP level.
Saving authentication information in the console also enables the site to be accessed from an Internet browser that was called from the home menu.

For details about automatic HTTP authentication, see the section entitled "Automatic HTTP Authentication Function."

3. 19 Getting the URL That Was Opened Last

When the Internet browser utility is shut down, the URL that was opened last during that session can be obtained.
Although this enables browsing results to be predicted to a certain degree, since there is no way to guarantee that the page has been opened using that URL, this function should not be used in such a way that the results are indispensable for game progress.

For details about how to set this item, refer to the descriptions of SceUtilityHtmlViewerParam and SceUtilityHtmlViewerBuffer in the Utility_InternetBrowser-Reference.

3. 20 Getting a Return Value From the Content

By defining a dedicated PSP™ plug-in and using it in JavaScript from the content, arbitrary data can be returned to the game program from the content.
This enables the content and game program to be linked. However, since there is no way to guarantee whether or not the page has been opened using the URL that was specified by the game program, if you use this function to control the progress of the game, you should implement a sufficient degree of protection required by the game program and content such as exchanging encrypted data.

For details about getting a return value from the content, refer to the section entitled "Content Linking Functions."

3. 21 Calling a Method From the Content

By defining a dedicated PSP™ plug-in and including JavaScript code in the content so it can be used, a method that was registered by the game program can be called from the content. This enables data to be exchanged in both directions between the content and the game program and allows a service that is provided by the game program to be used from the content. However, since there is no way to guarantee that the page was opened from the URL that was specified by the game program, if this feature will be used for some operation related to game progress, the data to be exchanged between the game program and the content should be suitably protected by a mechanism such as encryption.

For details about calling a method from content, see the chapter entitled "Content Linking Functions".

4 File Download Function

When the Internet browser utility is used to download a file from a website, operation will vary depending on the initialization parameters that were specified and the content description method used by the website. To obtain the desired operation, appropriate settings must be made and appropriate content must be created.
This section describes the standard file download operation and support for related operations.

4. 1 Standard Download Operation

The standard operation of the download function using the Internet browser utility is as follows.
  1. Confirm the start of downloading.
  2. Automatically determine the save destination directory according to a combination of the file name specified in the URL and the MIME-Type specified in the "Content-Type" HTTP response header.
  • If a file name is specified in the "Content-Disposition" HTTP response header, that file name takes precedence.
  • Some file formats (such as video data) will automatically change the file name in the Internet browser.
  1. Confirm the save destination.
  • Except for some formats (such as video data), the file name and save destination directory can be changed by the user.
  1. Start downloading.
  • During downloading, the progress is displayed.
  • If the user changed the save destination directory or file name on the downloading destination confirmation screen, then downloading will be performed using that save destination directory and file name.
  • If the required directory does not exist, it is automatically created.
  1. Confirm that downloading finished.

For information about the relationship between the file name and MIME-Type and the downloading destination directory, refer to the "Guidelines for Creating Content for the PSP™ (PlayStation® Portable) Internet Browser (for developer)" published on the PSP™ Developer Network website.

4. 2 Specification of the Directory and File Name by the Game Program

If you want to specify the save destination directory or file name from the game program, specify the directory name, the file name, or both in initialization parameters.
If the save destination directory or file name is specified, changes by the user are restricted, and specifications in the content are also disabled.
If you also want to specify the directory and file name of a file to be uploaded from the game program, specify the directory name and file name in initialization parameters.
If the directory and file name of the file to be uploaded are specified, the specified path is automatically entered in the form in the content, and changes by the user are restricted.

For details about how to set this item, refer to the description of SceUtilityHtmlViewerFileParam in the Utility_InternetBrowser-Reference.

4. 3 Specification of the File Name by the Content

If the file name is not specified in an initialization parameter, the file is downloaded using the file name that was specified in the URL of the content, but if the file name was specified by a CGI program using the "Content-Disposition" HTTP response header, for example, that file name takes precedence.
However, some formats (such as video data) automatically change the file name in the Internet browser.

4. 4 Restrictions on Changing the Save Destination Directory or File Name

If the save destination directory or file name is specified from the game program, changes to these items by the user are automatically restricted.
Otherwise, changes can be restricted by explicitly specifying "Do not display save destination confirmation dialog" or "Do not change save destination directory and file name" in initialization parameters.

4. 5 User Interface Restrictions

In addition to the "Do not display save destination confirmation dialog" and "Do not change save destination directory and file name" specifications, "Do not display confirmation dialog when downloading begins" and "Do not display confirmation dialog when downloading ends" can also be specified.

If "Do not display confirmation dialog when downloading begins" is specified, the save destination confirmation dialog is displayed immediately when the content to be downloaded is selected.

If "Do not display confirmation dialog when downloading ends" is specified, the downloading screen closes when downloading ends, and the screen returns to displaying the content.

If "Do not display save destination confirmation dialog" is specified, downloading begins immediately when the start of downloading is confirmed. The save destination is determined in a similar manner as when the save destination confirmation dialog is displayed.

If "Do not change save destination directory and file name" is specified, the save destination directory and file name items in the save destination confirmation dialog are grayed out and cannot be changed.

If "Do not display save destination confirmation dialog" is specified, no confirmation is performed before overwriting the file if it already exists. However, you can cause overwrite confirmation to be performed by specifying "Enable overwrite confirmation."

4. 6 Precautions

Automatically Determining the Save Destination Directories

5 Automatic HTTP Authentication Function

When browsing a website specified by the game program, HTTP authentication can be automatically performed without making the user enter a user name and password.
This enables access to the website from anyone other than the game program to be blocked without requiring any user intervention.
In addition, through a specification, the authentication information can be saved in the console enabling the website to be accessed from the Internet browser that is called from the home menu.
This section explains how the game program and content support the automatic HTTP authentication function.

5. 1 Authentication Information and Application Conditions

Authentication information to be applied to automatic HTTP authentication is specified from the game program.
The following authentication information is specified.
  • URL
  • User name
  • Password
  • Specification of authentication method and console saving operation
More than one set of this information can be specified.
Also, separately from the authentication information, specify the Realm (AuthName) of the website for which automatic HTTP authentication is to be applied. Only one Realm (AuthName) can be specified.

Automatic HTTP authentication is applied under the following conditions.
  • The Realm specification in the content begins with the Realm specified by the game program
  • The URL in the content begins with the URL specified by the game program
    (*) Authentication information that contains the longest matching URL is used.
  • The authentication method in the content matches the authentication method specified by the game program
Only if all of these conditions are satisfied will HTTP authentication be automatically performed and the target page displayed.

For details about how to specify authentication information, refer to the descriptions of SceUtilityHtmlViewerParam and SceUtilityHtmlViewerAuthInfo in the Utility_InternetBrowser-Reference.

5. 2 Support by Content

Server-side settings must enable HTTP authentication for the content. In addition, the following actions must be taken for all content for which automatic HTTP authentication is to be performed.
  • Set the authentication method to the one that matches what is used by the game program
    (*) Only BASIC authentication or DIGEST authentication can be used.
  • Set the Realm (AuthName) so that it is the same as the beginning of the string specified by the game program
  • Set the user name and password so that they match those of the game program
  • Specify the URL from the game program

For information about how to make these server settings, refer to the manual of the server software in use, or contact the server administrator.

5. 3 Saving Authentication Information

For each set of authentication information specified from the game program, you can specify whether or not the authentication information is to be saved in the console if authentication is successful.
If authentication information is saved, automatic HTTP authentication can be performed for the associated content even when the Internet browser is called from the home menu.
However, since the Realm (AuthName) cannot be saved, when authentication information is saved, the Internet browser must use the start of a specified default text string as the Realm (AuthName).
If the Realm (AuthName) does not match, automatic completion of the authentication information is performed as a normal operation. In other words, in that case, the user name that is contained in the authentication information is explicitly used.
Also, since authentication information in the console is automatically deleted beginning with the oldest existing information and since information may be intentionally deleted by the user, a game should not be designed based on the assumption that authentication information that was saved in the console is maintained.

For details about how to specify the saving of authentication information, refer to the description of SceUtilityHtmlViewerAuthInfo in the Utility_InternetBrowser-Reference.

6 Content Linking Functions

Content can be linked to a game program by defining a dedicated PSP™ plug-in and writing JavaScript code so that the plug-in can be used by the content to support the game program.
This section describes how to use these functions in the game program and in the content.

6. 1 Plug-ins and Linking Functions

The currently supported plug-ins and content linking functions that are provided are as follows.

PSP™ Plug-in
  1. Type (plugin identifer)
    application/x-psp-plugin
  2. Functions
  • Getting a return value from the content
  • Shutting down the Internet browser

PSP™ Extended Plug-in
  1. Type (plugin identifer)
    application/x-psp-extplugin
  2. Functions
  • Calling a method from the content
  • Calling a plug-in instance callback

To enable the linking functions with the PSP™ plug-in, the user interface must be set to limited mode or to no-user-interface mode. This is to prevent attacks by malicious content since the plug-in has the ability to shut down the Internet browser.

To use the linking functions, specify a dedicated PSP™ plug-in definition in the content as follows.

Example

 
<!-- for PSP Plugin -->
<object name="psp" type="application/x-psp-plugin"></object>
<!-- for PSP Ext-Plugin -->
<object name="pspext" type="application/x-psp-extplugin"></object>

Although the name attribute is set to "psp" or "pspext" in the example above, an arbitrary name can be specified. The object is handled in the JavaScript by using the name specified here.

For information about specifications in the content, refer to the "Guidelines for Creating Content for the PSP™ (PlayStation® Portable) Internet Browser (for developer)" published on the PSP™ Developer Network website together with this document.

6. 2 Getting a Return Value From Content

A return value is obtained from the content by having JavaScript in the content write data to a buffer provided by the game program.
The following example shows how to write a return value from the content.

Example

 
<script language="JavaScript">
<!--
a = 10;
b = new Array(3);
b[0] = 1; b[1] = 2; b[2] = 3;
if (psp) {
    psp.save(a);
    psp.save(b);
}
//-->
</script>

The save() method is used to write data. The data that is written will be a string with a format based on a JavaScript internal representation. In the example above, if "a" is used as the return value, "10" is written to the buffer, and if "b" is used as the return value, "1,2,3" is written to the buffer.
Also, if data is written multiple times while the Internet browser utility is active, the buffer is always overwritten by the data that was written last. In the example above, the b value will be valid.

For details about the acquisition method, refer to the descriptions of SceUtilityHtmlViewerParam and SceUtilityHtmlViewerBuffer in the Utility_InternetBrowser-Reference.

For information about specifications in the content, refer to the "Guidelines for Creating Content for the PSP™ (PlayStation® Portable) Internet Browser (for developer)" published on the PSP™ Developer Network website together with this document.

6. 3 Shutting Down the Internet Browser

The Internet browser can be shut down according to a specification from the content. This function can only be implemented in the content if the content linking functions are enabled.
The following example shows how to shut down the Internet browser from the content.

Example

 
<script language="JavaScript">
<!--
a = 10;
if (psp) {
    psp.save(a);
    psp.exit();
}
//-->
</script>

The exit() method is used to shut down the Internet browser. In the example above, "10" is written by the save() method as the return value and then the browser is shut down by the exit() method.

When the exit() method is executed, the behavior of the Internet browser is the same as if a shutdown operation were performed by the user.
In other words, if there is no particular specification otherwise, the shutdown confirmation dialog is displayed to prompt the user to select whether to shut down the browser or cancel the request. If you do not want to have a shutdown confirmation, specify "Do not display shutdown confirmation dialog" in an initialization parameter.
Also, the network connection will or will not be automatically disconnected on shutdown as specified in an initialization parameter.

For information about specifications in the content, refer to the "Guidelines for Creating Content for the PSP™ (PlayStation® Portable) Internet Browser (for developer)" published on the PSP™ Developer Network website together with this document.

6. 4 Calling a Method From the Content

A method call from the content is implemented by having the game program register a method with an assigned name, then calling the method using that name from JavaScript code within the content.
To call a method from the content, specify the code shown below.

Specification Example

 
<script language="JavaScript">
<!--
no = 10;
if (pspext) {
    ret = pspext.PlaySound(no);
    alert("ret="+ret);
}
//-->
</script>

"PlaySound," which is called in the above example, is a method that was registered by the game program. The method is called with a single argument, and it provides a return value. An integer type or string type can be specified by the game program for the return value.

To register a method in the game program, specify the following code.

Specification Example

 
static const SceUtilityHtmlViewerUserMethodInfo method_info[] = {
    }
        /* .plugin   = */ SCE_UTILITY_HTMLVIEWER_MIMETYPE_EXT_PLUGIN,
        /* .name     = */ "PlaySound",
        /* .callback = */ UserMethod_PlaySound,
        /* .usrdata  = */ (void*)0,
        /* .flag     = */ 0
    }
};

static const SceUtilityHtmlViewerParam param = {
    
    /* .num_method      = */ 1,
    /* .method          = */ method_info,
    /* .plugin_callback = */ NULL
};

static int UserMethod_PlaySound(int instance, int argc, char** argv,
                                void* usrdata,
                                unsigned int* resulttype, void** result)
{
    
    int no = atoi(argv[0]);
    if (no < 0 || no > MY_MAX_SOUND_NUM) {
        return -1;
    }
    int ret = MyPlaySound(no, );
    
    *resulttype = SCE_UTILITY_HTMLVIEWER_USER_METHOD_RESULT_TYPE_INT;
    *result = (void*)ret;
    return 0;
}

In the above example, UserMethod_PlaySound() is a callback function that corresponds to the method to be registered. When it is called, it performs processing according to the argument, sets the return value of the method, and returns.
When it is processed normally, 0 is set as the callback return value. If a non-zero return value is set, it is treated as an error, and the results of the method call are undefined.

For details about registering methods, refer to the descriptions of SceUtilityHtmlViewerParam, SceUtilityHtmlViewerUserMethodInfo, and SceUtilityHtmlViewerUserMethodCallback in the Utility_InternetBrowser-Reference.

For information about specifications in the content, refer to the "Guidelines for Creating Content for the PSP™ (PlayStation® Portable) Internet Browser (for developer)" published on the PSP™ Developer Network website together with this document.

6. 5 Plug-in Instance Callback Call

A plug-in instance callback is called when an instance of a dedicated PSP™ plug-in is created or destroyed.
Currently, only a PSP™ extended plug-in can be the target of a callback call.
A plug-in instance is created or destroyed at the following times.
  • When a page that contains the definition of a dedicated PSP™ plug-in is opened (created)
  • When a page that contains the definition of a dedicated PSP™ plug-in is closed (destroyed)
  • When a page that contains the definition of a dedicated PSP™ plug-in is reloaded (destroyed & created)
  • When a setting in a display menu is changed, such as when the character size is changed, in a page that contains the definition of a dedicated PSP™ plug-in (destroyed & created)

To register a plug-in instance callback in a game program, specify the following.

Specification Example

 
static const SceUtilityHtmlViewerPluginCallback plugin_callback = {
    /* .PluginInstanceNew     = */ OnInstanceNew,
    /* .PluginInstanceDestroy = */ OnInstanceDestroy,
    /* .usrdata               = */ 0
};

static const SceUtilityHtmlViewerParam param = {
    
    /* num_method      = */ 1,
    /* method          = */ method_info,
    /* plugin_callback = */ &plugin_callback
};

static int OnInstanceNew(const char* plugin,
                         void *instance,
                         int argc, char** argn char** argv, void* usrdata)
{
    
    
    return 0;
}

static int OnInstanceDestroy(const char* plugin,
                             void *instance,
                             int argc, char** argn char** argv, void* usrdata)
{
    
    
    return 0;
}

In the above example, OnInstanceNew() and OnInstanceDestroy() are callback functions corresponding to the creation and destruction of a plug-in instance. The instance identifier and attribute that was specified for that instance are obtained as arguments.
The instance identifier is a unique value that is assigned each time an instance is created. Since the same value is obtained as an argument even in a method call for that instance, the game program can switch processing for each instance.
The attribute, which is specified in the object tag in the content corresponding to that instance, also contains "name", "type" and corresponding values. By having the content dynamically change the attribute, the game program can change processing accordingly.
The callback return value is set to 0.

For details about registering instance callbacks, refer to the descriptions of SceUtilityHtmlViewerParam and SceUtilityHtmlViewerPluginCallback in the Utility_InternetBrowser-Reference.

For information about specifications in the content, refer to the "Guidelines for Creating Content for the PSP™ (PlayStation® Portable) Internet Browser (for developer)" published on the PSP™ Developer Network website together with this document.

7 Internet Browser Utility Restrictions

The Internet browser utility has restrictions on several functions.
This section explains those restrictions.

7. 1 Helper Application Function

Although an Internet browser that is called from the home menu has a function for executing helper applications for various types of content that were registered in advance, this function is partly disabled in the Internet browser utility.
Specifically, the following content cannot be used.
  • Some video distribution content
  • RSS channel XML data
