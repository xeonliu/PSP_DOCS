libgiq_picture Overview

1 Library Overview

1. 1 Functional Features

libgiq_picture is a lightweight version of libgim_picture. It provides an almost identical API as libgim_picture but can display GIM files with less CPU load.

1. 2 Related Files

The following files are required to use libgiq_picture.

  Category     Filename  
  Header file     giq/giq_picture.h  
  Library file     libgiq_picture.a  


  • libgu and libgum are required to draw in the standard context.
  • To use internal functions, be sure to include giq/giq_protected.h.
  • Source code is located in devkit/src/giq_picture/.

3 Restrictions

3. 1 Notes

  • Memory clusters, which are used for allocating groups of memory blocks, are now generated implicitly.

3. 2 Internal Data

  • Data structures were changed as follows.
  • Image and Palette blocks are registered in linked lists to improve performance.
  • Animation information is stored in the Sequence block.
  • Drawing data was changed as follows.
  • Texture configuration packets are created and used implicitly when possible.
  • Image pixel data is stored with 128-byte alignment.

4 Extended Functions

4. 1 Memory Manager

Additional functions that perform alignment and delayed release of memory have been implemented for the memory manager. These functions can be executed by the memory manager or by user-provided memory management functions as before.

4. 2 Memory Mutex

The memory manager can now be protected from access by multiple threads.
For information about thread-safeness, see Chapter 5.

4. 3 Picture Duplication

Pixel data can now be duplicated. This was not supported previously.

5 Thread-Safeness

5. 1 Thread-Safeness

libgiq_picture is designed so that pictures share internal data, consequently, libgiq_picture is not multithread safe.



Even if a picture is not duplicated and does not share internal data with another picture, some kind of access protection is still needed because memory manager state is stored in global variables.



To perform multithreaded processing, the following conditions must be satisfied.
  • A picture that is accessed by a sub-thread must not share internal data with another picture
  • The memory manager must be protected by a mutex

5. 2 File Loading

If file loading is executed in a sub-thread, for example, the picture that was just loaded will not share internal data with another picture. This is considered to be relatively easy to implement.

Consider the following sub-thread processing example.
  • File loading is performed asynchronously with respect to the main thread
  • The main thread is notified synchronously when file loading has completed



The following is an example of the sub-thread program.

void *data = 0 ;
size_t size = 0 ;
LoadFile( "dog.gim", &data, &size ) ;

SceGiqPicture *picture = sceGiqPictureCreate() ;
sceGiqPictureLoadFile( picture, data, size, 0 ) ;
sceGiqPictureLoadCache( picture ) ;
