GIM Format Overview


1 Introduction

1. 1 Overview

The 2D file format is used to store basic texture images. 2D format files can contain the following kinds of information:
  • Images
  • Palettes
  • Sequences

1. 2 Creation

2D format files are created by converting generic image files. BMP, TGA, TM2, DDS, AVI and PNG are the currently supported source file formats.


1. 3 Notational Conventions

In this document, data types are denoted as shown below. Byte order is Little Endian.

  Type Name     Type     Type Name     Type  
  char     Signed 8-bit integer     u_char     Unsigned 8-bit integer  
  short     Signed 16-bit integer     u_short     Unsigned 16-bit integer  
  int     Signed 32-bit integer     u_int     Unsigned 32-bit integer  
  float     32-bit floating-point            


The specified "alignment" is the minimum required value. For details, see "Restrictions Imposed by the PSP™ Library".

2 Storage Format

A 2D format file consists of a file header followed by image data arranged in a hierarchical structure of chunks.

2. 1 File Header

The file header contains information about the file itself. It has the following data structure.

typedef struct {
	u_int signature ;	// File identifier ( '.GIM' )
	u_int version ;		// Version number ( '1.00' )
	u_int style ;		// Data style ( 'PSP')
	u_int option ;		// Option (0)
} GimHeader ;

Data style is used to indicate that the data representation has been optimized for a specific API. The 'PSP' style indicates that this format is for use only with the PSP™ library. For more details, see "Restrictions Imposed by the PSP™ Library".

2. 2 Chunks

A chunk can store any type of data. Chunks can also have hierarchical structures. The layout of a chunk is shown below. The padding area is filled with zeroes.


The chunk header has the following data structure.

typedef struct {
	u_short type ;		// Chunk type
	u_short unused ; 	// Unused (always 0)
	u_int next_offs ;	// Offset of next chunk 
	u_int child_offs ;	// Offset of lower-level chunks
	u_int data_offs ;	// Offset of chunk data 
} GimChunk ;

The start of each element in a chunk must be aligned on a 4-byte boundary. As a result, all the offsets in a chunk header are multiples of 4. An offset is never 0 even if there is no corresponding element. Zeroes are added as padding bytes in order to align the data.

3 Chunk Structure

Chunks have the following hierarchical structure.

3. 1 File Chunk

A file chunk is the highest level chunk; it represents the entire file. A file chunk can contain multiple picture chunks, however, it does not contain any chunk data itself.

3. 2 Picture Chunk

A picture chunk represents the data for one picture. A picture chunk can contain the following kinds of chunks. The number of each chunk is arbitrary.
  • Image chunk
  • Palette chunk
  • Sequence chunk

A picture chunk does not contain any chunk data itself.

3. 3 Image Chunk

An image chunk can store an array of images; each image contains the same number of pixels. The array can be used as a mipmap or a sequence.



Mipmaps enable the number of pixels to be (n+1)/2 for each level. They also allow images with the same number of pixels to be stored in all levels.

The layout of image chunk data is shown below. Note that each area may be padded with zeroes at the end to ensure proper alignment.



Image Header
The image header has the following data structure.

typedef struct {
	u_short header_size ;	// Header size (48)
	u_short unused ;	// Unused (0)
	u_short format ; 	// Pixel format (GIM_FORMAT_XXXX)
	u_short order ;		// Pixel storage format (GIM_ORDER_XXXX)
	u_short width ;		// Width of images in pixels
	u_short height ;	// Height of images in pixels
	u_short bpp ;		// Number of bits per pixel
	u_short pitch_align ;	// Width alignment (number of bytes)
	u_short height_align ;	// Height alignment (number of pixels)
	u_short dim_count ; 	// Number of dimensions in image array (2)
	u_short reserved ;	// Reserved area (0)
	u_short reserved2 ;	// Reserved area (0)
	u_int offsets ;		// Offset to offset array
	u_int images ;		// Offset to first image
	u_int total ;		// Total size
	u_int plane_mask ;	// Plane mask
	u_short level_type ;	// Level type (GIM_TYPE_XXXX)
	u_short level_count ;	// Level count
	u_short frame_type ;	// Frame type (GIM_TYPE_XXXX)
	u_short frame_count ;	// Frame count	
} GimImageHeader ;

header_size indicates the header size. It must be at least 48 bytes.
format and order indicate the image format (details elsewhere).
width and height indicate the number of pixels in the images (min. 1).
bpp indicates the average number of bits per pixel (min. 1).
pitch_align and height_align indicate the units of alignment (min. 1).
dim_count indicates the number of dimensions in the image array (must be 2).
offsets is the offset into the offset array using the image header as the origin.
images is the offset to the first image using the image header as the origin.
total is the total size of the image chunk data.

plane_mask indicates the plane configuration when images having a small number of bits are stored so that they are stacked on top of each other. Bit sequences for each adjacent plane are inverted. The lowest level plane is set to 0 when the number of planes is even and 1 when the number of planes is odd. plane_mask is valid only when the format member specifies an index format. It should be set to 0 if it is not used.

Example:


level_count and frame_count indicate the number of images (min. 1)
level_type and frame_type indicate the type of image array.
Set level_type to either GIM_TYPE_MIPMAP or MIPMAP2.
Set frame_type to GIM_TYPE_SEQUENCE (details elsewhere).

User Data
Can be any arbitrary byte sequence.

Offset Array
Contains offsets to each image using the image header as the origin. The number of offsets is equal to the number of images. Offsets are four-byte integers and must be four-byte aligned.

Image
An image is an array of pixels. Images must be 16-byte aligned.

3. 4 Palette Chunk

A palette chunk can store an array of palettes; each palette contains the same number of entries. The array can be used as a mipmap or a sequence.



The layout of chunk data and the data structure for a palette chunk are basically the same as for an image chunk, however, the following restrictions apply.
  • height is always 1.
  • The only format is GIM_FORMAT_RGBAXXXX.
  • plane_mask is always 0.
  • The only level_type is GIM_TYPE_MIPMAP2.

3. 5 Sequence Chunk

Sequence chunks contain simple animation information such as:
  • Image frame number
  • Palette frame number
  • Image rectangle

Sequence chunks also contain constants used for various settings such as:
  • Alpha blending mode
  • Texture function
  • Texture filter
  • Texture wraparound

The layout of sequence chunk data is shown below. Each area must be four-byte aligned. Note that each area may be padded with zeroes at the end to ensure proper alignment.



Sequence Header
The sequence header has the following data structure.

typedef struct {
	u_short header_size ;	// Header size (32)
	u_short unused ;	// Unused (0)
	u_short frame_start ;	// Starting frame
	u_short frame_end ;	// Ending frame
	u_short frame_rate ;	// Frame rate
	u_short frame_repeat ; 	// Repeat mode(GIM_REPEAT_XXXX)
	u_short n_constants ;	// Number of constants
	u_short n_functions ;	// Number of functions
	u_int constants ;  	// Offset into constant data array
	u_int functions ;  	// Offset into function data array
	u_int keyframes ;  	// Offset to first keyframe sequence
	u_int total ;		// Total size
} GimSequenceHeader ;

header_size indicates the header size. It must be at least 32 bytes.
frame_start and frame_end indicate the playback range (in frames).
frame_rate indicates the playback speed (in frames/second).
frame_repeat indicates the repeat mode (described in detail later).
n_constants indicates the number of constants (>= 0).
n_functions indicates the number of functions (>= 0).
constants is an offset into the constant data array using the sequence header as the origin.
functions is an offset into the function data array using the sequence header as the origin.
total is the total size of the sequence chunk data.

User Data
Can be any arbitrary byte sequence.

Constant Data
Constant data is used to represent parameters that are not animated. Constant data has the following data structure.

typedef struct {
	u_short param ;		// Parameter (GIM_PARAM_XXXX)
	short value ;		// Value
} GimConstant ;

param specifies a parameter that is set with a constant value. value represents that value (details elsewhere).

Function Data
Function data represents parameters that are animated with keyframes. Function data has the following data structure.

typedef struct {
	u_short param ;		// Parameter (GIM_PARAM_XXXX)
	u_short interp ;	// Interpolation method (GIM_INTERP_XXXX)
	u_int keyframes ;	// Offset into keyframe sequence
} GimFunction ;

param specifies a parameter for which the value of the function is set. (details elsewhere).
interp indicates the method of interpolating between keyframes (details elsewhere).
keyframes is an offset into the keyframe sequence using the sequence header as the origin.

Keyframe
A keyframe represents the value of a function at a specific time. A keyframe has the following data structure. Consecutive keys are stored together.

typedef struct {
	short count ;		// Keyframe spacing
	short values[ n ] ; 	// Key values (variable number)
} GimKeyframe ;

If count is greater than or equal to 0, its value indicates the spacing to the next keyframe, and the values array will only contain one entry. If count is less than 0, it means that the number of consecutive keys is -count. In this case, the values array will contain -count entries.

Example:


count is set to 0 for the last keyframe.

3. 6 File Information Chunk

Information related to a file can be stored in a file information chunk.
The chunk data consists of multiple strings separated by NULL characters ('\0').

The meanings of the strings are defined as follows.
  • ProjectName (name of file for which information is created)
  • UserName (name of user who created file)
  • SavedDate (date and time file was created)
  • Originator (name of application that created file)
  • Comment (arbitrary)

5 Format Constants

5. 1 Pixel Format

GIM_FORMAT_INDEX4, INDEX8, INDEX16, INDEX32


GIM_FORMAT_DXTn
This is the S3TC texture compression format. In this format, pixels within the image are stored for each 4(4 block. For details, refer to documentation related to S3TC.
Note that the bpp that is contained in the image header is 1/16 of the total number of bits in one block. Also, pitch_align is 1/4 of the alignment that is required for 4 lines.
The memory layout within a block is shown below.

  • GIM_FORMAT_DXT1, DXT1EXT


  • GIM_FORMAT_DXT3, DXT3EXT


  • GIM_FORMAT_DXT5, DXT5EXT



5. 2 Pixel Storage Format

GIM_ORDER_NORMAL
This is the most generic pixel storage format. Pixels are stored sequentially from left to right and from top to bottom. Pixels less than 8 bits are stored from the low-order bits of the storage destination.


GIM_ORDER_PSPIMAGE
This pixel storage format can be loaded rapidly into the texture cache of the PSP™. Pixels are stored in blocks that are 16 bytes wide by 8 pixels high. Note that this format cannot be specified when the pixel format is S3TC.
