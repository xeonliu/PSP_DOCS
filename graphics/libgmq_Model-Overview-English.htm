libgmq_model Overview

1 Library Overview

1. 1 Functional Features

libgmq_model is a lightweight version of libgmo_model. It provides an almost identical API as libgmo_model but can display GMO files with less CPU load.

1. 2 Related Files

The following files are required to use libgmq_model.

  Category     Filename  
  Header file     gmq/gmq_model.h  
  giq/giq_picture.h  
  Library file     libgmq_model.a  
  libgiq_picture.a  


  • libgiq_picture is required to display textures.
  • libgu and libgum are required to draw in the standard context.
  • To use internal functions, be sure to include gmq/gmq_protected.h.
  • Source code is located in devkit/src/gmq_model/.

3 Restrictions

3. 1 Model Data Restrictions

  • Bones are stored in depth-first order
  • All vertex data used by a single mesh has the same vertex format

3. 2 Notes

  • Memory clusters, which are used for allocating groups of memory blocks, are now generated implicitly.
  • When a specific parameter is changed, the following functions must be called (for details, refer to the library reference).
  • sceGmqModelReconfigure()
  • sceGmqModelNormalize()

3. 3 Internal Data

  • Because data structures were changed, some APIs for accessing internal data are no longer supported or have become convenience functions for ease of use.
  • The changes in data structures are as follows.
  • The Arrays block has become a subsidiary block to the Mesh block.
  • To improve performance, the Arrays block is now registered in a linked list.
  • Material block information is stored in the Layer block.
  • The Animate command has been integrated into the FCurve block.
  • Drawing data was changed as follows.
  • Drawing primitives are expanded and stored in the drawing packet.
  • Index vertices are expanded and stored in non-index vertices.


4 Extended Functions

4. 1 Memory Manager

Additional functions that perform alignment and delayed release of memory have been implemented for the memory manager. These functions can be executed by the memory manager or by user-provided memory management functions as before.

4. 2 Memory Mutex

The memory manager can now be protected from access by multiple threads.
For information about thread-safeness, see Chapter 5.

4. 3 Model Duplication

Vertex and pixel data can now be duplicated. This was not supported previously.

4. 4 List Drawing

The bone drawing order can now be directly specified or sorted by Z value.

4. 5 Preset Drawing

The CPU load can now be reduced by creating packets in advance.

4. 6 Edge Drawing

Additional edge data can now be loaded from model data to selectively draw edges such as ridgelines.

4. 7 Sub-Block Names

Additional sub-block names can now be loaded from model data to perform name resolution for the animation target.

5 Thread-Safeness

5. 1 Thread-Safeness

libgmq_model is designed so that models share internal data, consequently, libgmq_model is not multithread safe.



Even if a model is not duplicated and does not share internal data with another model, some kind of access protection is still needed because drawing parameters and memory manager state are stored in global variables.



To perform multithreaded processing, the following conditions must be satisfied.
  • A model that is accessed by a sub-thread must not share internal data with another model
  • A sub-thread cannot call a drawing API for a model that it accesses
  • The memory manager must be protected by a mutex

5. 2 File Loading

If file loading is executed in a sub-thread, for example, the model that was just loaded will not share internal data with another model and drawing APIs will rarely be called. This is considered to be relatively easy to implement.

Consider the following sub-thread processing example.
  • File loading is performed asynchronously with respect to the main thread
  • The main thread is notified synchronously when file loading has completed



The following is an example of the sub-thread program.

void *data = 0 ;
size_t size = 0 ;
LoadFile( "dog.gmo", &data, &size ) ;

SceGmqModel *model = sceGmqModelCreate() ;
sceGmqModelSetOptionFlags( model, SCEGMQ_OPTION_IMPORT_PICTURES, 0 ) ;
sceGmqModelLoadFile( model, data, size, 0 ) ;
sceGmqModelLoadCache( model ) ;
The importing of external textures and generation of preset packets cannot be executed in the asynchronous part. When an external texture is imported, internal data is shared with another model. Similarly, when preset packets are generated, drawing APIs are called internally.

In the above example, to prevent external textures from being imported automatically, the file is loaded with the corresponding option flag cleared. To import external textures, sceGmqModelImportPictures() should be explicitly called in the main thread.
