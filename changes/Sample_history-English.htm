1. 2 devkit/sample/sound

In the sound samples under sound/, the function and symbol names that are used only in the samples and are not SCE library function/symbol names have been renamed as shown below.

  • The prefix of function names starting with sceMain was changed to the prefix "main".
  • From the symbol names for the callbacks, mutexes, event flags and threads within the samples, prefixes starting with SceMain have been deleted and the names have been changed to start with the root word.

Accordingly, those function/symbol names in the samples, that are not SCE library function/symbol names and are difficult to understand because SCE and sce prefixes are attached, have been reorganized.

1. 3 devkit/sample/sound/mp4aacplay
devkit/sample/sound/mp3play
devkit/sample/sound/at3mp3play
devkit/sample/sound/mp3lowlevel
devkit/sample/sound/aacplay

These sound samples performed file close/reopen when the media (UMD™, Memory Stick™) was ejected within the callback that was registered with the driver.
In this case, if a file on UMD™ is opened within the callback, the thread for which the callback was called will be blocked until the file open processing is completed.

To resolve this problem, the samples have been fixed so that the file close/reopen when the media is ejected are done in the BGM playback thread.

Note that in the mp4aacplay sample, do not place an MP4 file to play back other than on Memory Stick™ otherwise the sound will be interrupted.

3 Release 6.0.0

3. 1 sample/net/adhoc/adhoc_dialog

In releases earlier than 6.0.0, when using sceNetAdhocDialogJoin(), the timeout was determined when scan was performed 5 times by the user attempting to join the network. This processing time is actually about 5 seconds which was too short for timeout judgment. Especially, when using the adhocParty, it easily timed out.
As described in the "pspnet Overview" document, it is reasonable to extend the timeout to about 60 seconds.

In Release 6.0.0, this sample has been changed to perform scan 60 times.
Please be careful when referring to this samples that are an earlier release than 6.0.0.

If the network API returns an error when checking, follow the process below:

  1.     The API returns an error which cannot be ignored       
  2.     Check if an error event has been returned for the adhocctl event handler       
  3a.     If an error event occurred, the error event error code is displayed     -> terminated  
  3b.     If an error event did not occur, the API error code is displayed     -> terminated  

In addition, when using the Message Dialog Utility with SCE_UTILITY_MSGDIALOG_TYPE_NUMBER to display an error code, the Utility may terminate normally. In this case, there is no need for any response on the application side.
