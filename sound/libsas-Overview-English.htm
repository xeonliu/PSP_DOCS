PSP™ Programmer Tool Runtime Library Release 6.3.0

libsas Overview

1 Library Overview

1. 1 Functional Overview and Features

libsas is a library for controlling SAS. libsas enables waveform data such as ADPCM (.vag) to be used as phoneme data for performing functions such as waveform data sound generation, pitch change, envelope processing and digital effects on voice data.

Features

  • Controls the software audio synthesizer (SAS), which runs on the Media Engine
  • The output data format is 44.1kHz 16-bit 2-channel (with effects for stereo mode) or 4-channel output (without effects for multichannel mode)
  • The granularity (number of PCM data samples that are generated each time internal processing is executed) can be selected from 64 to 2048 in multiples of 32.
  • Supported waveform formats are ATRAC3™ (monaural only with no looping), ADPCM (.vag), and PCM (16-bit monaural)
  • Built-in noise sound source
  • Maximum number of simultaneously active voices is 32 (varies depending on process load.)
  • Built-in digital effects functions
  • Pitch can be controlled from the -12 octave to the +2 octave
  • Built-in ADSR mode envelope
  • Built-in pause function for a voice that is being vocalized

1. 2 SAS Features

As mentioned above, SAS runs on the Media Engine and is controlled from libsas. For details about audio synthesis processing performed by SAS (such as input/output data formats and internal audio processing), refer to the "SAS Overview."

2 Overview of Usage Procedure

2. 1 Procedure for Using libsas

The following procedure shows how to use libsas.
  1. Use sceSasInit() to perform initialization processing.
  2. Use sceSasSetVoice() to set the basic parameters for the voice for which sound is to be produced
  3. Use a function such as sceSasSetKeyOn() to perform various types of sound processing (such as key on processing).
  4. Use sceSasCore() to generate PCM sound based on key on information.
  5. Use sceWaveAudioWrite() to output the generated PCM sound
  6. Use sceSasExit() to perform termination processing.

2. 2 Initialization and Termination Processing

To use libsas, sceSasInit() must be used to perform initialization processing. Also, after using libsas, use sceSasExit() to perform termination processing when the application ends.

Example:
int iRc;

/* Initialization processing */
iRc = sceSasInit();
if (iRc != SCE_OK){
	/* Initialization failed */
}

/* Specify processing for using libsas functions here */

/* Termination processing */
iRc = sceSasExit();
if (iRc != SCE_OK){
	/* Termination processing failed */
}

2. 3 Method of Outputting PCM Sound That was Generated by sceSasCore()

With libsas, PCM sound is generated by the sceSasCore() function. The generated PCM sound is output by using the libwave function sceWaveAudioWrite(). To output PCM sound, first set information such as the output sample length.

Example:
int iRc;

/* Set channel 0 to stereo output */
iRc = sceWaveAudioSetFormat(0, SCE_WAVE_AUDIO_FMT_S16_STEREO);
if (iRc != SCE_OK){
	/* Setting failed */
}
/* Set the output sample length for channel 0 to 256 samples */
iRc = sceWaveAudioSetSample(0,256);
if (iRc != SCE_OK){
	/* Setting failed */
}

A function such as sceWaveAudioWrite() is used to output PCM-format sound. sceWaveAudioWrite() registers the PCM-format sound that is stored in the specified buffer in the audio driver. Playback of the registered sound data is started after sound data that had already been registered in the audio driver is played.
Audio is not necessarily played back immediately because it was registered in the driver. Also, the audio driver does not create a copy of the sound data. As a result, if the buffer contents are overwritten before playback is actually started or while playback is in progress, the contents are not played back correctly. Make sure that the sound output-only buffer is at least a double buffer so that the buffer contents immediately after sound is registered are not overwritten.

Example:
/* Let pcmBuf be a double buffer */
/* The buffer that is passed as an argument of sceSasCore() must have */
/* 64-byte alignment */
int iSide = 0;

sceSasCore( pcmBuf[iSide] );

/* Output 16-bit PCM STEREO data that is in pcmBuf[iSide] to channel 0 */
sceWaveAudioWrite(
	0,
	SCE_WAVE_AUDIO_VOL_MAX,
	SCE_WAVE_AUDIO_VOL_MAX,
	pcmBuf[iSide]
);
iSide ^= 1;

3 Linking With ATRAC3™

This chapter describes how to use ATRAC3™ in libsas and how it differs from ADPCM.
By using libatrac3plus together with libsas, an ATRAC3™ sound source can be used for the SAS phoneme region. Unlike ADPCM and PCM sound sources which only support on-memory playback, libsas also supports streaming playback for ATRAC3™, which can easily be performed.

3. 1 Overview of the Usage Procedure (for On-Memory Playback)

After SAS initialization is completed, ATRAC3™ data can be used in libsas according to the following procedure.
  1. Use a utility to load modules used by the libatrac3plus library.
  2. Read ATRAC3™ data.
  3. Use a function such as sceAtracSetDataAndGetID() to register ATRAC3™ data in libatrac3plus and obtain an ATRAC™ ID (handle for decoding).
  4. Use the phoneme region setting function sceSasSetVoiceATRAC3() to register the data that was registered for the acquired ATRAC™ ID for a specific voice of SAS.
Subsequently, the normal playback method is similar. sceSasSetADSR() and sceSasSetVolume() can be used to set playback parameters such as the volume and envelope shape, and the sceSasSetKeyOn() function can be used to begin sound production for the relevant voice.

Since the requirement for reading data is no different from that for a normal ADPCM voice, the only differences until the voice is vocalized are the following three.
  • In addition to libsas, modules related to libatrac3plus must be loaded.
  • An ATRAC3™ decoding handle is obtained before setting the phoneme region.
  • sceSasSetVoiceATRAC3() is used instead of the sceSasSetVoice() function.

One more difference is termination processing. For ADPCM or PCM data, you need not do anything in particular to destroy voice data that is no longer used, but for ATRAC3™ data, you must explicitly release the ATRAC™ ID. sceSasReleaseATRAC3Id() is used for this. You must also release the ATRAC™ ID if you want to register different voice data after you have registered ATRAC3™ voice data for a certain ATRAC™ ID.
Also, if you attempt to change the sound source of a voice for which the ATRAC3™ phoneme region is registered, an error will occur. Make sure you release the ATRAC™ ID first before changing the sound source.

An ATRAC™ ID that is registered for a SAS voice will be used exclusively by SAS and cannot be used by libatrac3plus until it is released with sceSasReleaseATRAC3Id(). In other words, the number of ATRAC™ IDs that can normally be used is reduced while one is being used by SAS.
Note that although the number of ATRAC™ IDs that can normally be used simultaneously by libatrac3plus is two for ATRAC3plus™ and two for ATRAC3™, since ATRAC3plus™ data cannot be used by SAS, the number of ATRAC3™ voices that can be used by SAS is at most two. If you need to use more than two ATRAC3™ voices, you can use the sceAtracReinit() function to reduce the number that can be used for ATRAC3plus™ data and increase the number that can be used for ATRAC3™ data. For details, refer to the libatrac3plus documentation.

3. 2 Overview of the Usage Procedure (for Streaming Playback)

Since it is assumed that relatively long-duration data will be played using ATRAC3™, libsas has a feature for supporting ATRAC3™ streaming playback. (However, libsas does not have fine-grained control of reading like libatrac3plus. It also does not support sound source data with looping.)
The ATRAC3™ voice generation method when the streaming support feature is used is roughly as follows.

  1. Read some ATRAC3™ data into buffer A.
  2. Use a function such as sceAtracSetDataAndGetID() to register ATRAC3™ data in libatrac3plus and obtain an ATRAC™ ID (handle for decoding).
  3. Use the phoneme region setting function sceSasSetVoiceATRAC3() to register the data that was registered for the acquired ATRAC™ ID for a specific voice of SAS.
  4. (Use a function such as sceSasCore() to synthesize and output a voice.)
  5. Look at the return value of sceSasCheckATRAC3BufferStatus() and wait until the other half of the double buffer can be safely overwritten.
  6. Read the continuation of the ATRAC3™ data into the other buffer (buffer B) of the double buffer.
  7. Use sceSasConcatenateATRAC3() to report that buffer B contains continuation data.
  8. Repeat (alternately switching buffer A and B) until the end of the file is reached.

To prevent the buffer that is currently playing from being overwritten and the data being destroyed, call sceSasCheckATRAC3BufferStatus() to check if it is safe to overwrite the data. Since the position of the buffer during playback changes when a voice synthesis function such as sceSasCore() is called, using sceSasCheckATRAC3BufferStatus() to check the buffer status is sufficient if it is only done once after the synthesis function call.
Also, since file reading normally takes time, sound might break up if a blocking function such as sceIoRead() uses the same thread to read data as the one playing sound. The asynchronous IO feature should be used instead.
Be careful about the buffer size when using streaming playback. In particular, since the amount of data required by the voice synthesis function at one time increases when a pitch change is performed to raise the pitch, the data required for voice synthesis may be insufficient. Also, to prevent a high rate of file reading, prepare a buffer that corresponds roughly to several seconds worth of data at normal playback speed (for example, approximately 64 KB for 66 kbps of data).

For details, see the sas-streaming sample program.

3. 3 Streaming Playback of a Stereo Voice

SAS only supports monaural ATRAC3™ files. As a result, to play a stereo voice, two voices must be assigned to the left and right channels and then sound production should begin for both voices simultaneously.
It is important that processing for both voices be performed simultaneously. In particular, when streaming playback is performed, since data reading may not be completed in time depending on the state of the media, if data for only one voice is replenished at this time, the subsequent playback positions will be displaced, and the sound will be of extremely poor quality. Be sure to replenish the data for both voices in this situation.

3. 4 Proper Use of libatrac3plus and libsas

Although libatrac3plus and libsas can both be used to perform streaming playback of ATRAC3™ data, the features they support differ with respect to the relationship between available memory and processing load.
Generally, libatrac3plus is better for playing several minutes or more of sound (such as BGM) without applying any particular effects, and libsas is better for playing and applying a variety of effects to voices (dialog, yelling, cheering, and some phrase-sampled tunes) when interactive changes are needed for periods from several seconds to several tens of seconds.

3. 5 Proper Use of PCM, ADPCM, and ATRAC3™ Data

Although PCM, ADPCM, and ATRAC3™ data can be used in libsas, the special characteristics of each and how best to use them are described below for your reference so that you can use them properly.

Since PCM data is an uncompressed sound source, it theoretically has the highest sound quality, but it is also has the largest footprint (approximately 86 KB per second at 706 kbps). A sound source that was saved in any compressed format can be decoded to PCM, which can be used by SAS, and voices synthesized by SAS can be fed back into SAS again (to obtain an echo-like effect). Due to size constraints, PCM data may often be difficult to use as raw phoneme region data.

ADPCM data has a format that is compressed by approximately 1/3.5 (approximately 25 KB per second at 202 kbps) compared to PCM. Since decoding is simpler, more voices can be played at the same time. It also has a relatively small size, making it suitable for use as phoneme region data such as for the voice of an ordinary musical instrument. Also, by using loops, data can be made even smaller in parts such as the reverberation of a musical instrument.

Since a higher compression rate is realized for ATRAC3™ data (approximately 8 KB per second at 66 kbps or 52 kbps), a longer interval of voice can be recorded with a smaller size. In other words, memory utilization and loading time can be improved. However, since a greater number of calculations and memory are required to play (decode) ATRAC3™ data compared to ADPCM, currently a maximum of six monaural voices can be played simultaneously with ATRAC3™.

You can balance data size and sound quality by using ATRAC3™ for voices that have a relatively long playback time and which need to produce any number of sounds simultaneously (fragments of phrase sampled BGM or sound effects such as cheering or yelling) and using ADPCM data for voices for which the data is short but the release rate is long and for which the playback of multiple voices overlap (such as the reverberation of a piano).

4 Precautions

4. 1 Important Precautions
Thread Priority Settings

Make the priority of the sound processing thread that uses a function such as sceWaveAudioWrite() to output PCM sound that was generated by sceSasCore() as high as possible. As long as there is no specific problem, setting SCE_KERNEL_USER_HIGHEST_PRIORITY is recommended. If the priority of the sound processing thread is low, a phenomenon may occur such as the generated sound being mixed with noise or the generated sound being intermittent or not even being played back at all.

4. 2 When Raising the Pitch

Although the maximum number of simultaneous sound-producing voices is 32, if the pitch is raised for many of the sound-producing voices, the playback sound could become intermittent. In this case, try not to raise the pitch very much (prepare a phoneme region that is recorded with increasing pitch) or try to decrease the number of simultaneous sound-producing voices.
In particular, note that the processing load will increase if ATRAC3™ data is used as the phoneme region.

4. 3 Using SAS Simultaneously With Other Processing

The software audio synthesizer (SAS), runs on the Media Engine. As a result, if other processing (such as playing movies or ATRAC3plus™, etc) that runs on the Media Engine is also executing at the same time, SAS sound may end up becoming intermittent. This is due to the fact that playing movies and ATRAC3plus™ decoding can take over the Media Engine for a long period of time.

One method of keeping SAS sound from becoming intermittent even if the conditions are such that SAS sound might be intermittent is to increase the granularity. Normally, sceSasCore() is executed once to generate 256-sample PCM data which is then output by one call to sceWaveAudioWrite(). However, the load can be reduced by changing this. Specifically, by executing sceSasCore() multiple times or by using the sceSasInitWithGrain() function to increase the number of samples per unit of granularity and then calling sceSasCore(), you can generate PCM sound for more than 256 samples and output that generated PCM sound for a greater number of samples by using a single sceWaveAudioWrite(). Since this increases the amount of PCM data that is output at one time and increases the interval in which that data is being output as sound, the result is that the sound is less likely to become intermittent. However, there is also the disadvantage that since the amount of PCM data that is generated at one time increases, the time until the next sound is produced increases.
Note that when sceSasInitWithGrain() is used to increase the granularity, the load will be lighter when compared to calling sceSasCore() multiple times with the granularity left unchanged, however, a delay or tempo discrepancy is more likely to occur.

4. 4 Parameter Update Timing

SAS uses batch processing to update internal parameters within the synthesis function (this function is called internally when a function such as sceSasCore() is called). As a result, the following are examples of the kinds of phenomena that occur.

(1) When sceSasSetVolume() is used to change the volume, the volume will not change immediately

After the sceSasSetVolume() function is called, the volume won't be changed until the next time the synthesis function is called. The volume of the sound occurring at the instant sceSasSetVolume() is called will not change.
This phenomenon is not limited to the volume setting, but also occurs for all key on, key off, and envelope shape changes.

(2) The value that is returned by the sceSasGetEndFlag() function is not immediately updated after a key on or key off function (sceSasSetKeyOn() or sceSasSetKeyOff() function) is called

The sceSasGetEndFlag() function is used to get the sound generation end flags for all voices, however, the flags are updated only once at the moment when the synthesis function call ends. Therefore, when multiple key on or key off operations are performed for the same voice before the synthesis function is called, you cannot use the end flags obtained from sceSasGetEndFlag() to verify that a voice is in an inactive state.

4. 5 Termination Processing

libsas internally calls the Media Engine's CPU functions and performs exclusive processing of the Media Engine's internal resources at that time.
Therefore, if sceKernelTerminateThread() is used to terminate a thread that is using libsas or if the thread is forcibly deleted by using sceKernelTerminateDeleteThread() before the thread has transitioned to DORMANT state, then depending on the timing, the thread may end up getting terminated and deleted while it is still holding internal resources. As a result, there is a risk that another library that is using the Media Engine's CPU functions, including libsas, may no longer operate properly.

To terminate a thread, make sure the following actions are taken (the samples that use libsas are also constructed in this way):
  • The side that requests the thread to terminate uses a shared variable or event flag to communicate its request to the thread.
  • The thread that is to be terminated periodically monitors this flag to determine whether a termination request was issued. If a request was issued, the thread first verifies that libsas is not currently running and then it calls sceKernelExitThread() to terminate itself.
