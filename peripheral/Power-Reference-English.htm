sceKernelPowerLock
 [Function]
Power off processing lock

Syntax

#include <kernel.h>
int sceKernelPowerLock (
	int locktype
);
Calling Conditions

Cannot be called from an interrupt handler
Can be called from a thread (must be called in a dispatch-enabled or interrupt-enabled state)
Multithread safe

Arguments

  locktype     Power processing lock type  
  SCE_KERNEL_POWER_LOCK_DEFAULT should be specified.  

Return Value

SCE_KERNEL_ERROR_OK
SCE_KERNEL_ERROR_ILLEGAL_CONTEXT
SCE_KERNEL_ERROR_CAN_NOT_WAIT
SCE_ERROR_INVALID_MODE

Description

This function delays a power off request until the lock is canceled.

It is used in a critical section to protect timing when a power interruption would cause a problem. For example, it might be used to prevent a power interruption while data is being written to the Memory Stick™.

Normally, the sceKernelPowerLock() function returns immediately. However, if it is called after suspend / standby processing has begun, the thread may be blocked.
Suspend / standby processing is performed by a thread within the power service. Therefore, if processing is completed when interrupts are disabled or dispatching is disabled, it is not necessary to enclose it between the sceKernelPowerLock() and sceKernelPowerUnlock() functions.

Serious Precaution
Since the user will not be able to turn the power off by operating the POWER/HOLD switch during a power locked state, if the sceKernelPowerLock() function is used carelessly, it can significantly decrease the usability of the PSP™.

For example, if a power lock is used when the PSP™ is waiting for an interaction from the user and the PSP™ is left in that state, the forced suspend function that causes the PSP™ to automatically transition to suspend state immediately before the battery runs out will not operate effectively.
Also, since the processing corresponding to operating the POWER/HOLD switch while the power is locked will be delayed until the lock is canceled by the sceKernelPowerUnlock() function, if the period for which the power is locked extends for a long time, to the user it will seem as though the power goes off quite unexpectedly.

The power lock must be used very carefully. Careless use should be sincerely avoided.

See Also

sceKernelPowerUnlock()

sceKernelVolatileMemLock
 [Function]
Get usage rights to the volatile memory by blocking

Syntax

#include <kernel.h>
int sceKernelVolatileMemLock (
	int arg,
	void **ppAddr,
	unsigned int *puiSize
);
Calling Conditions

Cannot be called from an interrupt handler
Can be called from a thread (must be called in a dispatch-enabled or interrupt-enabled state)
Multithread safe

Arguments

  arg     Specify SCE_KERNEL_VOLATILE_MEM_DEFAULT  
  ppAddr     Pointer to void * type variable for receiving starting address of volatile memory area  
  puiSize     Pointer to unsigned int type variable for receiving size of volatile memory area  

Return Value

If an error occurs, a negative value is returned.

  Value     Result  
  0     Processing was successful  
  < 0     An error occurred  


Description

This function enables the application to obtain the right to use the memory that is normally used for saving the contents of the eDRAM on the PSP™ system chip during suspend/resume processing.

This memory area is used by the utility library. However, by using this function, an application can obtain the right to use this memory area for exclusive processing, while the utility isn't running. This function operates by blocking. If you do not want blocking to be performed, use the sceKernelVolatileMemTryLock() function.
The usage rights that are obtained by the sceKernelVolatileMemLock() function are relinquished by the sceKernelVolatileMemUnlock() function.

Serious Precaution
It will not be possible to perform standby processing after using the sceKernelVolatileMemLock() function to obtain usage rights to the volatile memory. If SCE_POWER_CALLBACKARG_SUSPENDING or SCE_POWER_CALLBACKARG_STANDINGBY is detected by a callback that was previously registered with the scePowerRegisterCallback() function, you must quickly call the sceKernelVolatileMemUnlock() function to cancel the lock.

sceKernelPowerTick
 [Function]
Cancel idle timer

Syntax

#include <kernel.h>
int sceKernelPowerTick (
	int ticktype
);
Calling Conditions

Cannot be called from an interrupt handler
Can be called from a thread (must be called in a dispatch-enabled or interrupt-enabled state)
Multithread safe

Arguments

  ticktype     Specify the method of canceling the timer for measuring the time until power save mode begins.  

The following values can be specified for the ticktype argument.
  Macro     Description  
  SCE_KERNEL_POWER_TICK_DEFAULT     Cancels all timers.  
  SCE_KERNEL_POWER_TICK_SUSPENDONLY     Cancels only the timer related to automatic suspension.  
  SCE_KERNEL_POWER_TICK_LCDONLY     Cancels the timer related to the LCD.  


Return Value

Always returns 0.

Description

This function cancels the count value of the idle timer so that the PSP™ does not enter a power save state when the idle timer expires.

In idle state, the system automatically performs power save processing such as turning off the LCD backlight. Using the sceKernelPowerTick() function to cancel the count value for the idle state prevents the specified power save processing from being performed.

Specify SCE_KERNEL_POWER_TICK_SUSPENDONLY to cancel only the timer for beginning automatic sleep mode. However, in the current implementation, you do not need to use SCE_KERNEL_POWER_TICK_SUSPENDONLY because the automatic sleep function does not work while a game is running regardless of the PSP™ system software power save configuration.
Therefore, normally, only SCE_KERNEL_POWER_TICK_DEFAULT or SCE_KERNEL_POWER_TICK_LCDONLY should be used.


SCE_POWER_CALLBACKARG_STANDINGBY
 [Constant Definitions]
Standby status

Definition

#include <powersvc.h>
#define SCE_POWER_CALLBACKARG_STANDINGBY	0x00080000
Description

This constant indicates the bit for representing that standby state is in effect due to a POWER switch operation.

Sliding and holding the POWER switch (for at least two seconds) causes standby processing to begin. This bit is set for the callback that occurs after standby state begins.
If the power is turned off, then the next time it is turned on, the PSP™ system software will start up with a cold boot.

Precaution
If SCE_POWER_CALLBACKARG_STANDINGBY is detected when volatile memory is being used, you must call the sceKernelVolatileMemUnlock() function to relinquish the volatile memory usage rights.

When standby state is entered and SCE_POWER_CALLBACKARG_STANDINGBY is reported, the device driver begins processing required for interrupting the power supply. If an I/O access to the UMD™ or Memory Stick™ is attempted after this notification, it will fail. Also, since several seconds after this notification, the power will be automatically interrupted, the notification must not be used to cause the Memory Stick™ to be written immediately before the power is interrupted.

See Also

powercallback(), sceKernelVolatileMemUnlock()

SCE_POWER_CALLBACKARG_RESUMING
 [Constant Definitions]
Resuming status

Definition

#include <powersvc.h>
#define SCE_POWER_CALLBACKARG_RESUMING		0x00020000
Description

This constant indicates the bit for representing that resume processing has started.

It indicates that a resume operation has started because the POWER switch was operated in suspend state. This bit is set for the callback that occurs after the resume operation starts.

Precaution
When resume operation begins and SCE_POWER_CALLBACKARG_RESUMING is reported, the device driver begins processing for turning on the power supply again. If an I/O access to the UMD™ or Memory Stick™ is attempted after this notification but before SCE_POWER_CALLBACKARG_RESUME_COMP, processing may get blocked.

It takes several seconds from the time when resume processing starts until resume processing is entirely completed and SCE_POWER_CALLBACKARG_RESUME_COMP is reported in the callback.

See Also

powercallback()

SCE_POWER_CALLBACKARG_SUSPENDING
 [Constant Definitions]
Suspending status

Definition

#include <powersvc.h>
#define SCE_POWER_CALLBACKARG_SUSPENDING	0x00010000
Description

This constant indicates the bit for representing that suspend processing started.

It indicates that the POWER switch was operated for a short time (less than two seconds) or automatic sleep due to power save mode or forced sleep due to the battery running out occurred and suspend operation started. This bit is set for the callback that occurs after suspend operation starts.
A resume startup is performed the next time the power is turned on from suspend state. At this time, the PSP™ system software does not start up, but is restored to the state it was in when the suspend operation occurred.

Precaution
If SCE_POWER_CALLBACKARG_SUSPENDING is detected when volatile memory is being used, the sceKernelVolatileMemUnlock() function must be called quickly to relinquish the usage rights to volatile memory.

When suspend operation starts and SCE_POWER_CALLBACKARG_SUSPENDING is reported, the device driver begins processing for interrupting the power supply. If an I/O access to the UMD™ or Memory Stick™ is attempted after this notification, it will fail. Also, since several seconds after this notification, the power will be automatically interrupted, the notification must not be used to cause the Memory Stick™ to be written immediately before the power is interrupted.

The SCE_POWER_CALLBACKARG_SUSPENDING notification should only be used during a suspend or resume operation to automatically set pause state to prevent a game from advancing, or to stop movie streaming.

See Also

powercallback(), sceKernelVolatileMemUnlock()

scePowerRegisterCallback
 [Function]
Register power service notification callback

Syntax

#include <powersvc.h>
int scePowerRegisterCallback (
	int slot,
	SceUID uidCb
);
Calling Conditions

Can be called from an interrupt handler (does not depend on interrupt-enabled or -disabled state)
Can be called from a thread (does not depend on interrupt-enabled or -disabled state)
Multithread safe

Arguments

  slot     Callback registration slot  
  (0 to 15 or SCE_POWER_CALLBACKSLOT_AUTO)  
  uidCb     Callback UID  

Return Value

If an error occurs, a negative value is returned.

  Value     Result  
  0     Processing was successful (when slot is assigned a slot number of 0 to 15).  
  0 to 15     Processing was successful. Slot number was automatically assigned (when slot is set to SCE_POWER_CALLBACKSLOT_AUTO).  
  < 0     An error occurred.  


Description

This function registers a callback that is reported from the power service when a change in the POWER switch or battery status occurs.

If the sceKernelCreateCallback() function is used to create the callback in advance and the scePowerRegisterCallback() function is used to register the callback UID in the power service, then the callback is reported when a subsequent power-related status change occurs. When the thread that created a callback function enters a wait state with a callback, the relevant callback function is called in the context of that thread.
Up to 16 callbacks can be registered, and a callback is registered at the position specified by the slot argument. Only one callback UID can be registered per slot position.
If SCE_POWER_CALLBACKSLOT_AUTO is assigned to slot, an empty slot is automatically found, and the assigned slot number is returned by the function.
Specify this slot number when calling the scePowerUnregisterCallback() function to cancel the registration of the callback.

See Also

SCE_POWER_CALLBACKSLOT_AUTO, scePowerUnregisterCallback(),

Get maximum clock frequency for which WLAN can be used

Syntax

#include <powersvc.h>
int scePowerCheckWlanCoexistenceClock (
	void
);
Calling Conditions

Cannot be called from an interrupt handler
Can be called from a thread (must be called in a dispatch-enabled or interrupt-enabled state)
Multithread safe

Arguments

None

Return Value

If an error occurs, a negative value is returned.

  Value     Result  
  SCE_POWER_WLAN_COEXISTENCE_CLOCK_222MHz     Device type which permits a maximum PLL clock frequency of 222 MHz when the WLAN is in use  
  SCE_POWER_WLAN_COEXISTENCE_CLOCK_333MHz     Device type which supports PLL clock frequencies of 266 MHz and 333 MHz when the WLAN is in use   
  < 0     An error occurred  


Description

For the PSP-1000, the PLL clock frequency cannot be set to 266 MHz or 333 MHz while the WLAN is in use. However, for the PSP-2000 and later devices, the PLL clock frequency can be set to 266 MHz or 333 MHz while the WLAN is in use. Note that the behavior is different depending on the device type.

The device type on which the application is running can be checked in advance by calling scePowerCheckWlanCoexistenceClock(). Do not use scePowerCheckWlanCoexistenceClock() for any other purpose other than to obtain the maximum PLL clock frequency that can be set while the WLAN is in use.

With the PSP™ development tool DTP-T1000, boot parameter bit 11 can be selected on startup to cause the tool to operate either as a PSP-1000, or a PSP-2000 or later device. Be sure to verify that a program that uses PLL clock switching to set the PLL clock frequency to 266 MHz and 333 MHz operates properly with both settings of bit 11.

scePowerSetClockFrequency
 [Function]
Control clock frequencies

Syntax

#include <powersvc.h>
int scePowerSetClockFrequency (
	int iPllClock,
	int iCpuClock,
	int iBusClock
);
Calling Conditions

Cannot be called from an interrupt handler
Can be called from a thread (must be called in a dispatch-enabled or interrupt-enabled state)
Not multithread safe

Arguments

  iPllClock     PLL clock frequency [MHz]  
  Must be 190, 222, 266, or 333.  
  iCpuClock     CPU clock frequency [MHz] (1 to 333)  
  iBusClock     Bus clock frequency [MHz] (37 to 166)  
  (Must be exactly 1/2 of iPllClock.)  

Return Value

If an error occurs, a negative value is returned.

  Value     Result  
  0     Processing was successful  
  < 0     An error occurred  


Description

This function sets the PLL clock frequency, the CPU clock frequency and the bus clock frequency. The frequencies are specified in MHz.

For the PSP-1000, during WLAN operation, the PLL clock frequency cannot be set to either 266 MHz or 333 MHz. However, for the PSP-2000 and later devices, the PLL clock frequency can be set to 266 MHz or 333 MHz while the WLAN is in use. Note that the behavior is different depending on the device type.The device type on which the application is running can be checked in advance by calling scePowerCheckWlanCoexistenceClock().

Also, do not set the PLL clock frequency to a value less than 190 MHz.
The bus clock frequency must be set to exactly 1/2 of the PLL clock frequency.

The following table shows the valid combination of iPllClock, iCpuClock, and iBusClock to support operation in the current version.

  iPllClock     iCpuClock     iBusClock  
  333     1 - 333     166  
  266     1 - 266     133  
  222     1 - 222     111  
  190     1 - 190     95  


Note that even if the clock frequencies are set by using the scePowerSetClockFrequency() function, the clocks will not necessarily be set to those frequencies. The clock frequencies that are actually set can be obtained by using the scePowerGetPllClockFrequencyFloat(), scePowerGetCpuClockFrequencyFloat(), and scePowerGetBusClockFrequencyFloat() functions.

The scePowerSetClockFrequency() function changes the frequencies of the PLL clock, CPU clock, and bus clock. Note that the scePowerSetCpuClockFrequency() function changes the frequency of the CPU clock only, without changing the PLL clock frequency.
Note that the CPU clock and bus clock frequencies that are actually generated may not be exactly the same as the values set by using the scePowerSetCpuClockFrequency() function. This is because the CPU clock and bus clock are derived from the PLL clock, so depending on the value of the PLL clock frequency set by the scePowerSetClockFrequency() function, some difference may occur during clock generation.

Note
It may take some time to change the PLL clock frequency. Be sure not to change the clock frequency in situations when it is not permitted, such as when doing so would result in a processing failure because interrupts would be disabled for a long time. In particular, when the PLL clock frequency is changed, all bus masters (GE, VFPU, Memory Stick™, UMD™, WLAN, USB, Media Engine) and not just the CPU, will be paused. Also, do not change the PLL clock frequency when doing so would cause sound to be intermittent due to a drop in processing.

Note that the time required to execute the scePowerSetClockFrequency() function may vary depending on the device type and number of devices.

scePowerSetCpuClockFrequency
 [Function]
Control CPU clock frequency

Syntax

#include <powersvc.h>
int scePowerSetCpuClockFrequency (
	int iCpuClock
);
Calling Conditions

Cannot be called from an interrupt handler
Can be called from a thread (must be called in a dispatch-enabled or interrupt-enabled state)
Not multithread safe

Arguments

  iCpuClock     CPU clock frequency [MHz] (1 to 333)  

Return Value

If an error occurs, a negative value is returned.

  Value     Result  
  0     Processing was successful  
  < 0     An error occurred  


Description

This function sets the CPU clock frequency in units of MHz.

Although the ALLEGREX CPU normally operates at the CPU clock frequency, it switches to operate at the bus clock frequency while the load/store unit of the CPU is performing a bus access.

Note that even if the CPU clock frequency is set by using the scePowerSetCpuClockFrequency() function, it does not necessarily operate at that frequency. The CPU operating clock frequency that is actually set can be obtained by using the scePowerGetCpuClockFrequencyInt() or scePowerGetCpuClockFrequencyFloat() functions.

The PSP™ kernel has a special thread called the idle thread, which runs when all other threads are in a WAIT state. However, a special instruction for stopping the CPU clock is executed by the idle thread. From a power consumption viewpoint, the scePowerSetCpuClockFrequency() function need not be used to manipulate the CPU clock frequency if threads are appropriately set in a WAIT state so that the PSP™ transitions as quickly as possible to the idle thread when there is no task that must be processed.

It takes some time to change the clock frequency. Be sure not to change the clock frequency in a situation when it is not permitted to do so, such as when there is a drop in processing because interrupts have been disabled for a long period of time.
